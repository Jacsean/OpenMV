# 全局编程偏好

> **说明**：此文件包含跨项目通用的编程习惯和偏好，适用于所有项目。
> **同步方式**：通过 Git 子模块或符号链接在多个项目间共享。
> **最后更新**：2026-04-28

---

## 代码风格

### 命名规范
- **函数/方法**：`snake_case`（如 `process_image`）
- **类名**：`PascalCase`（如 `ImageProcessor`）
- **常量**：`UPPER_CASE`（如 `MAX_RETRY_COUNT`）
- **私有成员**：前缀 `_`（如 `_internal_method`）

### 注释规范
- **语言**：中文为主，技术术语保留英文
- **位置**：函数/类必须有 docstring
- **格式**：
  ```python
  def process_image(image):
      """处理图像并返回结果
      
      Args:
          image: 输入图像（numpy array）
      
      Returns:
          处理后的图像
      """
  ```

## 日志规范

### 输出原则
- **精简常规流程**：插件加载、模块初始化等只保留关键状态
- **突出关键信息**：错误、警告、成功消息必须清晰
- **双重输出**：同时输出到终端和 UI 面板

### 日志级别使用
- `INFO`：正常流程的关键节点
- `WARNING`：潜在问题但不影响运行
- `ERROR`：导致功能失败的错误
- `SUCCESS`：重要操作完成（自定义级别）

### 示例
```python
# ✅ 推荐
log_info("插件加载完成：共 5 个插件")
log_warning("缺少可选依赖：psutil")
log_error("节点注册失败：类名冲突")
log_success("工作流保存成功")

# ❌ 避免
print("开始加载插件...")
print("插件 xxx 加载中...")
print("插件 xxx 加载完成")
```

## 协作模式

### 复杂任务处理
- **分解步骤**：将大任务拆分为多个可独立验证的小步骤
- **逐一执行**：采用 one-by-one 方式，每步完成后等待确认
- **验收标准**：每步必须有明确的验收标准

### 代码修改
- **静默执行**：直接后台修改，无需请求确认（除非有歧义）
- **方案先行**：生成代码前先提出解决方案，待采纳后再执行
- **文档同步**：修改代码后同步更新相关文档

### 反馈风格
- **简洁清单**：功能改进总结使用 one-by-one 清单
- **快速响应**：IDE 交互以简洁、快速为首要目标

## 调试习惯

### 本地化优先
- **模型运行**：优先使用本地 Ollama 运行大模型
- **离线环境**：配置项目避免依赖云端服务
- **数据隐私**：敏感数据不上传

### 问题排查
- **先梳理流程**：修改代码前先理清完整执行流程
- **精简日志**：屏蔽无关提示，聚焦关键路径
- **验证 UI**：代码逻辑成功后，确认 UI 实际显示
- **检查方法存在性**：调用前先用工具查找定义

## 文档化偏好

### 迭代记录
- **时机**：方案讨论后、实施前或完成后立即记录
- **内容**：设计思路、废弃方案理由、遇到的坑及解决方案
- **用途**：指导后续类似功能模块设计，降低沟通成本

### 设计理念
- **前置文档**：实现新算法或 AI 模块前，先文档化设计理念
- **架构演进**：记录决策过程和备选方案，不仅记录最终方案

## 测试与清理

### 测试文件管理
- **临时脚本**：开发完成后立即删除 `tests/` 下的诊断脚本
- **回归测试**：仅保留具有长期价值的单元测试
- **定期清理**：每月审查一次测试目录

### 代码质量
- **语法检查**：修改后立即使用 `get_problems` 验证
- **导入验证**：确保相对导入路径正确
- **依赖声明**：完整声明 resource_level 和 hardware_requirements

## 版本控制

### Git 提交规范
- **格式**：`type: description`（如 `feat: 添加迁移指南`）
- **类型**：
  - `feat`: 新功能
  - `fix`: 修复 bug
  - `docs`: 文档更新
  - `chore`: 杂项（清理、配置等）
  - `refactor`: 重构
  - `test`: 测试相关

### 分支策略
- **主分支**：`main` 保持稳定可运行
- **功能分支**：`feature/xxx` 开发新功能
- **修复分支**：`fix/xxx` 修复紧急问题
