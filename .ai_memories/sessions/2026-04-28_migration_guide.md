**标签**：#架构 #记忆管理 #跨平台

## 背景

用户询问如何在重装操作系统或更换设备时，保留 AI 助手的三层记忆（会话/项目/全局）。核心需求是：
1. 全局记忆能在不同设备间同步
2. 项目记忆随代码提交到 Git
3. 会话记录保存在项目中形成开发历史
4. **关键要求**：用户可以浏览和编辑这些记忆，AI 助手也能随时读取使用

---

## 讨论要点

### 1. 现有记忆系统的局限性

- **会话记忆**：仅存在于当前对话窗口，关闭即消失，无法迁移
- **项目记忆**：依赖 AI 助手的内部存储机制，可能因路径变化失效
- **全局记忆**：通常存储在用户配置目录，重装系统后可能丢失

**问题**：
- ❌ 不透明：用户无法直接查看记忆内容
- ❌ 不可控：依赖外部同步机制（如云端账户）
- ❌ 不可移植：不同设备/系统间迁移困难

### 2. 解决方案对比

#### 方案 A：依赖 AI 助手的内置同步
- ✅ 优点：无需额外配置
- ❌ 缺点：黑盒、不可控、依赖服务商

#### 方案 B：导出配置文件手动备份
- ✅ 优点：完全掌控
- ❌ 缺点：操作繁琐、易遗漏、版本管理困难

#### 方案 C：**基于文件的记忆系统**（✅ 推荐）
- ✅ 优点：
  - 透明：Markdown 文件可直接查看编辑
  - 可控：通过 Git 版本控制
  - 可移植：随代码一起迁移
  - 团队共享：克隆即可见
- ❌ 缺点：需要手动维护（但可通过工具简化）

### 3. 目录结构设计

```
.ai_memories/
├── global/           # 全局记忆（跨项目通用）
├── workspace/        # 项目记忆（当前项目特定）
└── sessions/         # 会话记录（开发历史）
```

**设计理由**：
- 分离关注点：三种记忆类型独立管理
- 灵活性：全局记忆可通过子模块共享
- 可追溯性：会话记录按日期归档

---

## 决策结果

### 采用方案 C：基于文件的记忆系统

**核心原则**：
1. **文件化**：所有记忆以 Markdown 格式存储
2. **Git 管理**：纳入版本控制，支持团队协作
3. **透明可编辑**：用户可直接修改 [.md](file://d:\example\projects\StduyOpenCV\README.md) 文件
4. **AI 可读**：助手通过 `read_file` 工具读取作为"记忆"

**实施细节**：

#### 全局记忆
- **位置**：`.ai_memories/global/coding_preferences.md`
- **同步方式**：Git 子模块或符号链接
- **内容**：编程习惯、协作模式、调试偏好等

#### 项目记忆
- **位置**：`.ai_memories/workspace/project_specifications.md`
- **同步方式**：直接提交到项目 Git
- **内容**：架构决策、插件规范、UI 标准等项目特定内容

#### 会话记录
- **位置**：`.ai_memories/sessions/YYYY-MM-DD_topic.md`
- **同步方式**：直接提交到项目 Git
- **内容**：重要讨论、设计决策、问题解决过程

## 经验教训

### 成功经验

1. **文档优于记忆**：将规范写入文件比依赖 AI 内部记忆更可靠
2. **透明性至关重要**：用户应能直接查看和编辑"记忆"
3. **Git 是终极备份**：版本控制系统天然支持迁移和协作

### 遇到的坑

1. **过度设计**：最初考虑复杂的同步机制，后来简化为纯文件方案
2. **路径依赖**：不同操作系统的路径差异需要考虑（Windows vs Linux）
3. **权限问题**：符号链接在某些系统上可能需要特殊权限

### 改进方向

1. **自动化工具**：开发 CLI 工具简化记忆管理
2. **模板系统**：提供标准化的记忆模板
3. **搜索优化**：建立全文索引便于快速查找
