## 🔧 技术实现细节

### 1. 坐标系统设计

```
Qt场景坐标 (QGraphicsScene)
    ↑ 映射
视图坐标 (QGraphicsView)
    ↑ 转换
屏幕坐标 (globalPos)
```

**关键点**:
- 鼠标事件获取的是全局坐标
- 通过 `mapFromGlobal()` 转换为视图坐标
- 通过 `mapToScene()` 转换为场景坐标
- 标注存储使用场景坐标（与图像绑定）

### 2. 渲染架构

```python
display_image() 被调用时:
    1. 清空场景
    2. 添加原始图像 (QPixmap)
    3. 调用 redraw_annotations()
       - 遍历所有可见标注
       - 在场景上绘制图形项
```

**优势**:
- QGraphicsItem自动处理缩放/平移
- 无需手动计算变换矩阵
- 性能良好（Qt底层优化）

### 3. 工具状态管理

```python
current_tool = None | 'rect' | 'circle' | 'text'
drawing_start_pos = None | QPointF
```

**状态机**:
```
空闲 → 选择工具 → 鼠标按下 → 拖动(预览) → 释放(完成) → 空闲
```

## 💡 设计决策记录

### 为什么选择QGraphicsScene而非QPainter？

**决策**: 使用QGraphicsScene + QGraphicsItem

**理由**:
1. ✅ 自动处理缩放/平移变换
2. ✅ 每个标注是独立对象，易于管理
3. ✅ Qt底层优化，性能良好
4. ❌ QPainter需要手动计算坐标变换

### 为什么标注存储在场景坐标而非像素坐标？

**决策**: 使用场景坐标（相对于图像左上角）

**理由**:
1. ✅ 与图像绑定，缩放时自动调整
2. ✅ 便于导出为通用格式
3. ❌ 像素坐标需要在缩放时重新计算

### 为什么不使用独立的Overlay层？

**决策**: 直接在QGraphicsScene上绘制

**理由**:
1. ✅ 简化架构，减少复杂度
2. ✅ QGraphicsItem天然支持分层（z-value）
3. ❌ 独立层需要额外的同步逻辑

## 🎓 经验总结

### 成功经验
1. **模块化设计**: 将标注逻辑独立为单独模块，便于维护和测试
2. **先写测试**: 单元测试先行，确保数据模型正确
3. **渐进式实施**: 按Phase分阶段，避免一次性实现过多功能

### 遇到的问题
1. **导入路径冲突**: `ui/__init__.py` 导入了其他模块，导致测试困难
   - **解决**: 使用 `importlib` 直接加载模块
2. **方法重复定义**: 编辑过程中不小心复制了 `display_image` 方法
   - **解决**: 仔细检查代码结构，删除重复部分

### 改进建议
1. 考虑使用logging模块替代print语句
2. 添加更详细的tooltip和状态提示
3. 考虑添加快捷键支持（如Delete删除选中标注）
