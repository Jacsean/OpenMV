# 全局代码审查规则（所有提交必须通过）

## 1. 审查目标
每次修改后必须检查：**正确性、可读性、可维护性、安全性、性能、兼容性**

## 2. 必查项（逐条检查）
### ✅ 功能正确性
- 逻辑是否符合需求
- 正常流程/异常流程是否覆盖
- 边界值（空、0、负数、极大值）是否处理
- 是否有未定义行为、硬编码、魔法数字

### ✅ 代码质量
- 命名是否清晰、一致
- 注释是否完整、准确
- 重复代码是否提取（DRY）
- 函数/类是否单一职责
- 嵌套、长度、参数是否超标

### ✅ 安全漏洞
- SQL注入、XSS、CSRF
- 未授权访问、越权操作
- 敏感信息泄露（日志、返回、前端存储）
- 依赖漏洞、版本风险
- 输入未校验、未过滤

### ✅ 性能问题
- 循环内IO/查询
- 全表扫描、无索引
- 大对象未释放、内存泄漏
- 同步阻塞、无缓存
- 不必要的计算、重复请求

### ✅ 可维护性
- 配置与代码分离
- 日志规范、异常友好
- 无死代码、无无用导入
- 格式统一、可lint
- 文档/注释与代码同步

### ✅ 兼容性
- 版本兼容（语言/框架/依赖）
- 环境兼容（Windows/Linux、开发/生产）
- 接口兼容（新旧版本、字段变更）

## 3. 审查流程
1. 先看整体结构、命名、注释
2. 再走一遍逻辑：正常 + 异常 + 边界
3. 重点查：安全、性能、重复代码
4. 最后：格式化、lint、无warning
5. 输出审查报告：问题 + 建议 + 修复方案

## 4. 禁止项
- 禁止提交：warning、console.log、TODO、硬编码密钥
- 禁止提交：未校验输入、未处理异常、未释放资源
- 禁止提交：重复代码、超长函数、过深嵌套
